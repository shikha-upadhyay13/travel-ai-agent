Section 01

What is Agentic UI? — Design Philosophy

Agentic UI is a design paradigm where the AI agent drives the interface — not the user. Instead of the user navigating menus, filling forms, and making decisions at every step, the agent asks, the agent fetches, the agent presents, and the agent acts. The user's role shifts from "operator" to "decision-maker."

Traditional UI vs. Agentic UI

TRADITIONAL UI (Form-Based) AGENTIC UI (Conversation-Based) User opens app User opens app │ │ ▼ ▼ Sees 6 input fields: Agent: "Where do you want to go?" From ___ To ___ Date ___ │ Class ___ Quota ___ Passengers ___ ▼ │ User: "Tirupati, next Friday" ▼ │ User fills each field manually ▼ │ Agent extracts: ▼ from: Hyderabad (auto-detected) Clicks "Search" to: Tirupati │ date: April 4 (next Friday) ▼ class: ? (will ask) Sees list of 50 trains │ User scrolls, compares, confuses ▼ │ Agent: "Sleeper or AC?" ▼ User: "Sleeper" Clicks one train │ │ ▼ Sees another form: Agent: "Found 3 trains. Best one Passenger 1 Name ___ departs 6AM, reaches 11AM, ₹340. Age ___ Gender ___ Book this one?" Berth ___ User: "Yes" │ │ ▼ ▼ Fills form → Clicks "Pay" Agent: "Done! Ticket booked. ✅ │ PNR: 4521678901. Need a hotel?" ▼ Redirected to payment gateway │ ▼ Comes back → "Booking confirmed" 15+ steps, 20+ form fields 5 conversational turns Requires literacy + tech skill Requires only speaking ability User drives the interface Agent drives the interface

The core insight: An agentic UI doesn't just put a chatbot on top of forms. It fundamentally reimagines the interaction model. The form still exists in the backend — but the agent fills it, not the user. The user only makes decisions.

Section 02

The 7 Laws of YatraAI's Agentic Design

Law 1: Conversation is the Interface

Every feature is accessible through conversation. There are no hidden settings, no nested menus. If a user says "cancel my Delhi ticket", the agent handles it. The chat IS the app.

Law 2: The Agent Always Leads

The user should never wonder "what do I do next?" The agent always ends with a question or a suggestion. After booking, it proactively asks about stays. After showing results, it suggests the best option. It's never passive.

Law 3: Show, Don't Form

Never show a blank form. Show rich cards with real data that the user taps to select. Train results are cards, not table rows. Payment is a bottom sheet with one tap, not a form with 6 fields. Seats are visual grids, not dropdown menus.

Law 4: One Decision at a Time

Never overwhelm with 50 results. Show the best 3. Never ask 4 questions at once. Ask one. The agent manages complexity so the user doesn't have to.

Law 5: Language is Invisible

The user never "selects a language from a dropdown." The agent detects language from the first message and mirrors it. If someone types in Telugu, everything — responses, cards, buttons, confirmations — is in Telugu.

Law 6: Voice and Text are Equals

The voice experience is not a degraded version of the text experience. Voice gets the same agent intelligence, the same booking flow, the same proactive suggestions. It's just a different modality of the same agent.

Law 7: Transparency Builds Trust

The agent always says what it's doing. "Searching trains..." "Connecting to IRCTC..." "Payment of ₹580 received!" Users never face a blank loading state with no explanation.

Section 03

Agent Personality & Voice Design

Agent Character Profile

Name: YatraAI (no personified name — it's a helpful assistant, not a character)

Tone: Warm, respectful, efficient. Like a trusted travel agent at the local railway station who knows every train, speaks your language, and treats every customer with dignity.

NOT: Overly casual / slang-heavy / robotic / corporate / condescending / over-enthusiastic

Cultural markers: Uses "🙏" greeting (appropriate across India). Uses respectful forms ("aap" not "tum" in Hindi). Acknowledges religious/pilgrimage travel with respect ("Tirupati darshan ke liye? Bahut accha!").

Section 04

Conversation Flow Patterns

Pattern 1: Progressive Slot Filling

PROGRESSIVE SLOT FILLING Agent fills required "slots" one at a time through conversation Required Slots for Train Booking: ┌──────────┬──────────┬──────────┬──────────┬──────────┐ │ FROM │ TO │ DATE │ CLASS │PASSENGERS│ │ ____ │ ____ │ ____ │ ____ │ ____ │ └──────────┴──────────┴──────────┴──────────┴──────────┘ Turn 1: User: "मुझे कल दिल्ली जाना है" Slots filled: FROM=auto(HYD) TO=Delhi DATE=tomorrow Missing: CLASS, PASSENGERS Turn 2: Agent: "किस class में? Sleeper, 3AC, या 2AC?" User: "3AC" Slots filled: CLASS=3AC Missing: PASSENGERS Turn 3: Agent: "कितने लोग जाएंगे?" User: "मैं अकेला" Slots filled: PASSENGERS=1(self) ALL SLOTS FILLED → Trigger Search ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ KEY RULE: If user provides multiple slots in one message, the agent doesn't re-ask. It fills what it can and asks only for what's missing. Example of efficient fill: User: "Book Rajdhani Delhi to Mumbai tomorrow 2AC for 2 people" Slots filled: ALL → Skip directly to passenger details

Pattern 2: Confirmation Before Action

CONFIRM → ACT → REPORT The agent ALWAYS confirms before spending money or making irreversible actions Agent: "यह रहा आपका booking summary: 🚆 Rajdhani Express (12437) 📍 Hyderabad → Delhi 📅 7 April, 6:15 PM → 6:45 AM 💺 3AC · 1 passenger 💰 ₹2,100 Confirm करें?" Quick Reply Chips: [✅ Confirm & Pay] [✏️ Change] [❌ Cancel] User taps "Confirm & Pay" → Payment sheet slides up User taps "Change" → Agent asks "What do you want to change?" User taps "Cancel" → Agent says "Koi baat nahi, aur kuch help?"

Pattern 3: Proactive Suggestion After Completion

PROACTIVE POST-BOOKING FLOW Booking confirmed │ ▼ Agent: "✅ Booking done! E-ticket भेज दिया।" │ ▼ Agent proactively checks: ├── Is destination a pilgrimage city? → Suggest dharamshalas ├── Is travel date > 1 day? → Suggest hotels ├── Is there a connecting journey? → Suggest bus/flight ├── Is user a senior citizen? → Remind about wheelchair service └── Is arrival time late night? → Suggest waiting room/retiring room Agent: "दिल्ली में रुकना है तो ये options हैं: 🙏 Gurudwara Bangla Sahib (Free) 🏨 Retiring Room (₹350) 🏢 OYO Karol Bagh (₹899)" The agent never stops at "booking done." It anticipates the next need.

Pattern 4: Graceful Disambiguation

HANDLING AMBIGUOUS INPUT User: "मुझे बैंगलोर जाना है" Agent detects: TO=Bangalore, but missing everything else BAD (Traditional chatbot): "Please enter: From station, Date of journey, Class, Number of passengers" → This is just a form in chat form. Not agentic. GOOD (Agentic): "बैंगलोर! 👍 कब जाना है — कोई specific date?" → One question. Warm. Natural. ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ User: "Bangalore" Which Bangalore? There are 2 stations. BAD: "Select station: 1. KSR Bengaluru (SBC) 2. Yeshwanthpur (YPR)" → User may not know station codes GOOD: "Bangalore City (main station) ya Yeshwanthpur? Agar pata nahi toh main city station laga deta hoon 👍" → Offers a smart default. Doesn't demand the user know codes.

Pattern 5: Multi-Turn Correction

USER CHANGES MIND MID-FLOW Agent: "Sleeper class mein 3 trains mili..." User: "Actually, AC mein dikhao" BAD: "Please start a new search with different parameters" GOOD: "AC mein dekhta hoon..." → Re-searches with class=AC → Shows new results without losing other context (date, route) ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ Agent: About to book Rajdhani 3AC... User: "Wait, ek aur passenger add karo — meri wife bhi jayegi" GOOD: "Wife ka naam aur age batayiye, unko bhi add karta hoon" → Agent adds to passenger list without restarting booking flow

Voice Mode UI/UX Design

Voice Interaction Flow

VOICE MODE — COMPLETE UX FLOW ENTRY POINTS FOR VOICE ├── In-App Microphone → Tap 🎙️ in chat → Push-to-talk ├── Call Agent Button → Tap 📞 in header → Initiates voice call ├── External Phone Call → Dial +91-XXXX → IVR → Agent └── WhatsApp Voice Note → Send voice message → Transcribed → Reply ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━ PHONE CALL IVR FLOW User dials +91-XXXX-XXXXXX │ ▼ Auto-greeting (2 seconds): "YatraAI mein aapka swagat hai. Section 10

Error Handling & Recovery Patterns

AGENTIC ERROR HANDLING — EVERY ERROR IS A CONVERSATION PRINCIPLE: Never show a generic error screen. Always speak through it. 1. API TIMEOUT Traditional: ⚠️ "Error 504: Gateway Timeout" Agentic: "IRCTC thoda slow chal raha hai abhi. 2 options hain: 1. 30 seconds wait karein 2. Main baad mein try karke WhatsApp pe bataunga" [Wait] [Notify me later] 2. NO TRAINS FOUND Traditional: "No results found. Try different dates." Agentic: "Is date pe koi train nahi hai 😔 Lekin: • Kal (7 Apr) ko 5 trains hain • Bus se 8 ghante mein pahunch sakte hain (₹450) • Flight bhi hai ₹2,800 mein Kya dekhna hai?" [Tomorrow trains] [Bus today] [Flight] 3. PAYMENT FAILED Traditional: "Payment failed. Transaction ID: XYZ123" Agentic: "Payment nahi hua — koi baat nahi! Aapke paise deducted nahi hue hain ✅ Dobara try karein ya doosra UPI use karein?" [Retry GPay] [Try PhonePe] [Cancel booking] 4. MISUNDERSTOOD INTENT Traditional: "I didn't understand. Please rephrase." Agentic: "Hmm, mujhe lagta hai aap train booking chahte hain. Kya yeh sahi hai? • Hyderabad → Delhi • 6 April" [Haan, sahi hai] [Nahi, kuch aur] 5. BOOKING WAITLISTED Traditional: "Your ticket is waitlisted (WL 15)" Agentic: "Ticket waitlisted hai (WL 15). Chances: ~70% confirm hoga 🤞 Options: 1. Book karein — auto-cancel if not confirmed 2. Tatkal try karein kal 10 baje 3. Doosri train dekhein" [Book WL] [Remind for Tatkal] [Other trains]
